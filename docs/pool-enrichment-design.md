## Summary

Goal: every pool account shows both its **plan** (e.g. `plus`) and its **subscription expiry date**. Both live in the codex token's JWT claim `https://api.openai.com/auth` (`chatgpt_plan_type`, `chatgpt_subscription_active_until`), and that claim is only populated when the account was authorized through OpenAI's `/oauth/authorize` leg with three enrich params. There is exactly one working reference for those params in-repo: `server/cliproxy/internal/auth/codex/openai_auth.go:71-82` (`GenerateAuthURL`), which always sends `scope="openid email profile offline_access"`, `id_token_add_organizations=true`, `codex_cli_simplified_flow=true` against `client_id=app_EMoamEEZ73f0CkXaXp7hrann` (verified by adversarial pass 1).

Two onboarding modes:

- **Mode A (default) — CLIProxyAPI enriched login.** Interactive, human-in-the-loop per account. Produces a natively-enriched `auth.json` (plan + date in the JWT). Honest ceiling: one OpenAI hosted-consent screen per account, no headless/password grant → fine for tens, not thousands.
- **Mode B (bulk) — go-pool import.** go-pool (external, user-controlled) must add the same three params to its own authorize request **and** use an OpenAI-registered `redirect_uri` for that client_id. Then its exported JSON is enriched natively and the existing in-repo import + display path works. go-pool must **re-authorize** each account — existing lean tokens cannot be upgraded by refresh.

The token-exchange leg and the browser authorize leg both region-gate on OpenAI; Cloudflare is a non-issue because the chosen design never calls the Cloudflare-protected subscription endpoints (subscription rides inside the JWT). The unrecoverable gap is the **subscription date of any account genuinely authorized lean** (never enriched) — it exists nowhere in a lean file/token and refresh cannot backfill it. Note the important nuance folded in below: an account that was *enriched at login but had its `id_token` wiped by the pre-fix refresh bug* is a **different** population and may already be recovered by shipped code — see "Existing lean accounts".

## Mode A — CLIProxy enriched login

**What actually enriches.** Only the `/oauth/authorize` leg carries the params, and only `GenerateAuthURL` (`openai_auth.go:66-86`) sends them. Two entry points funnel through it and are confirmed enriched: the CLI flag `--codex-login`, and the management REST endpoint `GET /v0/management/codex-auth-url` (`RequestCodexToken`, `auth_files.go`, which pass 1 confirmed calls `GenerateAuthURL`). The device-code flow `--codex-device-login` is **[unverified]** — see Honest limits #3 (prior evidence is contradictory). The token-exchange body (`openai_auth.go:107-113`) is structurally just `grant_type/client_id/code/redirect_uri/code_verifier` — no room for enrich params on that leg, which is why enrichment can only happen at authorize time.

**Two callback/region facts that dictate the topology:**
1. `redirect_uri` is the hardcoded constant `http://localhost:1455/auth/callback` (`openai_auth.go:28`, sent at both authorize and exchange). Overriding `--oauth-callback-port` only moves where the *local* listener binds; the URL OpenAI redirects to is always `localhost:1455`. So the machine running the browser must be able to reach `localhost:1455` on the cliproxy host (same host, or an SSH tunnel).
2. OpenAI region-checks **both** the browser authorize screen and the server-side token exchange. The server-side exchange uses `CodexAuth.httpClient`, which honors `cfg.ProxyURL` (`NewCodexAuthWithProxyURL`, `openai_auth.go:48-60`) → set `proxy-url` in `config.yaml` to a supported-region (e.g. US) proxy and the exchange exits there. The browser leg is **not** the cliproxy httpClient, so the global proxy does not cover it — the browser itself must be in-region (or behind its own proxy).

**Recommended integration — A2 (login where the browser is, then import the JSON).** Run `cli-proxy-api --codex-login` (or `--no-browser` to paste the callback URL back) on a machine already in a supported region (or behind a US proxy set in that machine's `config.yaml`). Complete OpenAI consent once in a real browser. The enriched `auth.json` is written to that machine's local `auth-dir`. Then paste it into the target pool through the existing single-paste path: frontend `addPoolAuthFile` → `POST /api/pool/auth-files` → `AddPoolAuthFile` (`controller/xju_pool_auth.go:119-167`). `unwrapPoolAuthContent` (`xju_pool_auth.go:173-200`) passes a bare codex object through **byte-for-byte** and cliproxy's `writeAuthFile` persists the exact bytes (pass 1 traced this end-to-end) — so the enriched JWT survives untouched and `extractCodexIDTokenClaims` reads plan + date directly. This decouples the region/callback problem from claude-tri entirely and needs **no new endpoint**.

**Operational caveat (new).** A2 introduces a **third machine** that handles a live OpenAI bearer credential (`id_token`/`access_token`/`refresh_token`) outside the project's two-machine secret posture (claude-vps builds, claude-tri deploys). That machine must be trusted at the same level as claude-tri; the pasted JSON transits the pool-page textarea → new-api backend, so treat it as a secret in transit and scrub it from that machine's disk/clipboard/shell history after the paste. This is an operational requirement, not a code change.

**Alternative — A1 (login on claude-tri via the management endpoint).** Drive `GET /v0/management/codex-auth-url?is_webui=1`; with `is_webui` a `callbackForwarder` binds `:1455` on the cliproxy host and relays to the main server's `/codex/callback` route, which the handler goroutine polls, then exchanges and saves. Requirements: (a) `proxy-url` in that pool's `config.yaml` pointed at a US proxy so the token exchange passes the region check; (b) because tri is headless and `redirect_uri` is hardcoded `localhost:1455`, the operator must SSH-tunnel `-L 1455:localhost:1455` from an in-region browser machine to complete the consent screen. This lands the JSON directly in the target pool's `auth-dir` (no separate import), but adds the tunnel + per-pool proxy setup. A2 is simpler; A1 only wins if you want the account to land straight into a specific tri pool without a paste step.

**Cloudflare:** not encountered. Mode A only hits `/oauth/authorize` and `/oauth/token` (region-gated, not Cloudflare-gated). The subscription value comes from the returned JWT, so we never touch the Cloudflare-403 endpoints (`backend-api/{subscriptions,accounts/check,me}`).

**Does not scale — stated honestly.** Every account needs a human to pass OpenAI's hosted consent screen once; there is no password grant or scripted headless login for this client. Mode A is for the handful of accounts you can log in by hand; bulk goes through Mode B.

## Mode B — go-pool enrichment

**Exact change go-pool must make.** go-pool constructs and calls its own OpenAI `/oauth/authorize` URL. Make it match `GenerateAuthURL` (`openai_auth.go:71-82`) by adding to the authorize query string:

- `scope=openid email profile offline_access` — `offline_access` yields the refresh token; it is **not** what drives enrichment but must be present to match the working reference and keep the refresh token.
- `id_token_add_organizations=true` — one of the two enrich drivers.
- `codex_cli_simplified_flow=true` — the other enrich driver.

Keep everything else identical to the working reference so the subsequent code→token exchange succeeds: `client_id=app_EMoamEEZ73f0CkXaXp7hrann`, `response_type=code`, PKCE with `code_challenge_method=S256`, `prompt=login`.

**`redirect_uri` constraint (new, critical).** OpenAI validates `redirect_uri` against the URIs *registered* for `client_id app_EMoamEEZ73f0CkXaXp7hrann`. Matching go-pool's own exchange `redirect_uri` is necessary but **not sufficient** — an unregistered value fails the authorize leg (`invalid_redirect`) regardless of the enrich params, and no enriched token is produced. The known-good registered value is cliproxy's `http://localhost:1455/auth/callback` (`openai_auth.go:28`). go-pool should use exactly that (or another value it can prove is registered for this client).

**Export shape constraint (new).** The in-repo import path extracts **only `accounts[0]`** of an exporter bundle: `unwrapPoolAuthContent` (`xju_pool_auth.go:180-181`) reads `obj["accounts"].([]any)` and takes `accounts[0]` only, and the zip importer applies the same per-entry unwrap. Therefore go-pool must emit **one account per file** (single bare codex object, or a bundle whose `accounts` array holds exactly one account) — a single multi-account bundle silently imports 1 account and drops the rest; a zip whose entries are each multi-account bundles loses N−1 per entry.

With the three params the returned `id_token`'s `https://api.openai.com/auth` claim carries `chatgpt_plan_type` **and** `chatgpt_subscription_active_until`; go-pool's exported account JSON then embeds them natively.

**Once enriched, in-repo works unchanged.** A fully-enriched go-pool JSON imports through the existing zip path (`importPoolAuthFiles` → `POST /api/pool/auth-files/import` → `ImportPoolAuthFiles`), which stores each entry as-is (only bundle-wrapper unwrapping). `extractCodexIDTokenClaims` (`auth_files.go:623-661`) then reads plan + date **straight from the JWT** — this path needs the P2 patch for **nothing** when the token is fully enriched. The frontend already renders both fields (`web/src/features/pool/index.tsx:664-665`, `api.ts:52-53`), so **no frontend change is needed for display** (pass 1 confirmed).

**Re-authorize, not retrofit.** The three params only apply at the `/oauth/authorize` leg. Existing lean accounts already hold their tokens; you cannot upgrade them in place: the OpenAI token endpoint **rejects** `id_token_add_organizations` on a refresh grant (HTTP 400 `unknown_parameter`, live-verified this session), and `RefreshTokens` sends only `scope: "openid profile email"` with no enrich path. So **go-pool must re-run the full authorize flow per account** to get an enriched token. Accounts already farmed lean stay lean until re-authorized.

**Login-time exit IP (open, out of this repo's scope).** The one-`proxy-url`-per-pool model below governs **cliproxy's** server-side token exchange only. go-pool runs entirely outside cliproxy; at bulk scale it may need many different login-time exit IPs to avoid OpenAI fraud/rate-limit detection, and go-pool's own token-exchange proxy configuration is **unspecified** here. Flag it for go-pool's operator.

## Existing lean accounts (default + k12 pools)

**Reconciling the repo's two contradictory diagnoses (new, load-bearing).** Commit b21c986 (today, on `main`) says blank plan/subscription was caused by the **refresh bug wiping `id_token`**; the session memory note `pool-account-enriched-oauth.md` says that diagnosis "是错的" (is wrong) and the real cause is the **OAuth enrich-params difference** (go-pool lean vs CLIProxy enriched). Both are partially right, for **different populations**:

- **Population 1 — genuinely lean (go-pool, never enriched).** The `https://api.openai.com/auth` claim in *both* `id_token` and `access_token` lacks plan/subscription; plan survives only as a **top-level file field**, and the date exists **nowhere**. b21c986's access_token fallback does **not** help these (the access_token is also lean). This is the memory note's observed default-pool population (4 go-pool accounts). **PLUS badge: recoverable via P2** (top-level plan). **Date: NOT recoverable in place** — only re-authorization through an enriched flow puts a date on the account.
- **Population 2 — enriched at login, then refresh-wiped.** These *did* carry plan+date; the pre-fix refresh wiped the `id_token`. For these, b21c986's **access_token JWT fallback can recover plan AND date without re-auth — IF the currently-stored access_token still carries the enriched claim.** Whether a *post-refresh* access_token retains the enriched claim is **[unverified]**, and the HARD-CONSTRAINT observation "a plain refresh_token grant keeps tokens lean" suggests a refreshed access_token may itself be lean, in which case there is nothing left to recover. Going forward, Fix A's `!= ""` guard preserves the original enriched `id_token` across refreshes, so *newly* enriched accounts keep their date.

**Practical stance:**
- Run the classification audit (In-repo #5) to find out how many blank accounts are Population 1 vs 2. The design's earlier blanket "date is never recoverable in place" is corrected to: *never recoverable for Population 1; possibly already recovered by shipped code for Population 2.*
- Ship P2 to light up every genuinely-lean account's **plan** badge immediately; treat the **date** for Population 1 as a re-onboarding outcome, not a backfill.
- **k12 reality check (repo-grounded):** the k12 pool's ~500/501 accounts are reported basically dead as of 2026-07-15 (~500 at `402 deactivated_workspace`, per memory `k12-pool-batch-import.md`). Backfilling their subscription date is therefore largely **moot** — re-onboarding effort should target live accounts, not the dead k12 batch.

## Region / Cloudflare / proxy

- **Token-exchange leg (server-side):** `config.yaml` `proxy-url` (global) is honored by `CodexAuth.httpClient` (`openai_auth.go:48-60`) and applies to OAuth token-exchange and upstream traffic. Set it to a supported-region socks5/http/https proxy for any pool doing Mode-A login on tri, or for enriched-go-pool token exchange if go-pool ran on tri.
- **⚠️ Template gap (new):** neither `deploy/config.example.yaml` nor `deploy/config.k12.example.yaml` contains a `proxy-url` field (verified). Since dynamically-created pools clone `config.k12.example.yaml`, every one-click pool would silently lack a region proxy unless an operator hand-edits its live (gitignored) `config.yaml` on tri. Recommendation: add a commented `# proxy-url: ""` placeholder to **both** templates plus a runbook checklist entry, so the mitigation has a version-controlled landing spot.
- **Authorize browser leg (Mode A):** `redirect_uri` is hardcoded `localhost:1455`; the global proxy does not cover the browser. Handle by running the browser in a supported region (A2), or SSH-tunnel `:1455` to an in-region browser (A1).
- **Upstream per-account routing:** each auth JSON may set `proxy_url` (overrides global; `"direct"`/`"none"` bypasses) for fine-grained geo-exit per account. This is **post-login upstream routing only** — do not conflate it with the login-time region gate, which is governed by the pool's global `proxy-url` (server exchange) and the browser's own network (authorize).
- **Cloudflare:** designed out. Both modes obtain the subscription from the JWT at authorize time; neither hits the Cloudflare-protected `backend-api/{subscriptions,accounts/check,me}` endpoints (the two RULED-OUT approaches).

## Honest limits

- **Never gets a subscription date:**
  1. **Genuinely-lean accounts not re-authorized** (Population 1) — the date exists nowhere; refresh can't add it (400); active fetch is 403. Only a fresh enriched authorize recovers it. (Population 2, enriched-then-wiped, is a separate case — possibly already recovered by b21c986's access_token fallback; see "Existing lean accounts".)
  2. **go-pool accounts if go-pool is not patched** — remain lean; P2 gives plan-only, no date.
  3. **`--codex-device-login` accounts — [unverified], evidence is contradictory.** The specific structural mechanism previously asserted (a `deviceauth/usercode` call taking only `client_id`) is **not grounded in this repo** — that endpoint string appears nowhere in `server/cliproxy`, and `DoCodexDeviceLogin` routes through the *same* codex provider SDK (`manager.Login(ctx,"codex",…)`) that `--codex-login` uses, so the branch is inside the SDK and unverifiable from these files. Prior session memory (`pool-account-enriched-oauth.md`) explicitly states both `--codex-login` **and** `--codex-device-login` "都走 enriched." So device-login **may be enriched** and, if so, would be a materially *better*, less-interactive Mode A. Treat as **[unverified]** and confirm live before relying on it either way; do not assert it is lean.
  4. **Accounts OpenAI simply doesn't populate** — a free/no-subscription account has no `chatgpt_subscription_active_until` even when enriched; the badge can show plan but the date is legitimately absent.
- **Subscription date is a snapshot, not a live value (new).** The date is baked into the token JWT at enrichment time. A *real* subscription change afterward (renewal, upgrade, cancellation) is **not** reflected in the stored token until the account is re-authorized — refresh does not re-enrich. Downstream, the displayed date can therefore be stale relative to the account's true current subscription state.
- **Frontend date value-type is [unverified] and fragile (new).** `api.ts:52` types `chatgpt_subscription_active_until` as `string` and `index.tsx:103` parses it with `new Date(raw)`, but the Go claim field is `any` and is passed through unnormalized. Empirically it is an ISO-8601 string today (kaylahill renders correctly; tests use `'2027-06-01T00:00:00Z'`). If OpenAI ever emitted a **numeric Unix epoch (seconds)**, `new Date(number)` would read it as milliseconds → render as Jan-1970 and `isSubscriptionExpired` (`index.tsx:107-110`) would flag a valid account as "expired". Recommendation: pin the wire type (assert ISO-8601), and either normalize to ISO on the Go side in `extractCodexIDTokenClaims` or guard numeric input in `subscriptionUntil`. Low priority (no current mis-render), but cheap insurance.
- **Mode A does not scale:** one human consent per account; no headless/password grant for this client. Bulk must be Mode B (patched go-pool).
